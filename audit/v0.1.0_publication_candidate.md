## Source identity and supersession

Git verified that the audited baseline and original Phase 1 commit are ancestors of the replacement
candidate. The working tree and index were clean. The only configured remote was canonical `origin`,
whose fetch and push URLs both resolved to `https://github.com/MrithunjoyB/quantforge-ai`.

The former candidates were superseded after public runners exposed three genuine
portability/workflow defects:

1. Windows Server 2022 lacks the POSIX-only `os.fchmod` API. Secure atomic writes now retain
   descriptor-level `0600` hardening where available and otherwise rely on `tempfile.mkstemp`'s
   private exclusive creation plus the inherited Windows ACL. A regression simulates a platform
   without `fchmod`. Repair commit: `371206c09951856eab45545890605477d6c82416`.
2. The exact-SHA workflow correctly verified its checkout but the validator rejected the normal
   GitHub checkout remote. Offline no-remote validation remains the default; remote validation now
   accepts only one canonical GitHub HTTPS `origin` whose fetch and push URLs match an explicit
   expected repository. Mismatched and additional remotes have regression coverage. Repair commit:
   `95349b724c9291bbf47ad2595e004470b7d32949`.
3. Once atomic writes succeeded, a unit test still compared Windows' synthesized POSIX mode bits to
   an exact POSIX `0600` value. The exact `0600` assertion remains mandatory on POSIX; Windows now
   verifies its meaningful owner read/write representation while retaining the atomic, symlink,
   cleanup, and round-trip controls. Repair commit:
   `50c73f8a1bc6c7abea64edcf8a2f50e5e3dd2dec`.

No history was rewritten, amended, rebased, force-pushed, or deleted. No tag, GitHub Release,
release asset, GitHub Package, or PyPI publication exists.
